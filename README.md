# MyProt
# 通用协议网关底层原理简述

网关核心是一个**纯配置驱动的协议执行器**，所有通信行为由 JSON 配置文件定义，引擎仅负责解释执行，不含任何特定协议的硬编码。

---

### 1. 请求报文构建

操作定义中包含 `requestTemplate`（字符串列表）。构建请求时，逐行处理：

- **固定十六进制**：如 `"03 00"` 直接转为对应字节。
- **变量占位符**：`{变量名:格式}`，从标签的 `variables` 字典中取值，并按格式（如 `X2`、`X4`）转换为大端十六进制字节。
- **内置函数**：
  - `{TransactionID:auto:X4}` – 自动递增 2 字节事务标识符。
  - `{Length:calc:X4}` – 计算该字段之后所有字节的总长度，并填入 2 字节。
  - 可扩展 CRC、LRC 等校验计算。

所有字节段最终拼接为完整请求 `byte[]`。

---

### 2. 通信通道与帧解析

通道抽象 TCP 和串口，统一提供 `SendReceiveAsync` 方法。根据协议配置的 `framing` 决定如何从字节流中切分完整响应帧：

- **LengthField**：先读取固定头部，根据长度字段（偏移、字节数、是否含帧头、字节序）计算总帧长，再读取剩余内容。
- **Fixed**：直接读取指定固定长度的字节。
- **Raw**：由调用方显式指定期望长度（通常用于握手步骤）。

帧解析完成后获得完整响应 `byte[]`。

---

### 3. 响应验证与数据提取

- **有效条件检查**：若操作定义了 `validCondition`（如 `"resp[7] == 0x03"`），引擎解析表达式并验证响应中对应字节的值。
- **数据提取**：若配置了 `responseParser`，则根据 `dataStartIndex` 和 `dataLengthExpr`（支持动态表达式如 `"resp[8]"`）截取数据段，返回原始字节或按指定 `valueType` 转换为具体类型。若无解析器，则直接返回完整响应帧。

---

### 4. 并发控制

每个物理设备通道内部持有一个 `SemaphoreSlim(1,1)`，所有对该设备的请求必须先获取锁，响应完成后释放。这保证了：

- 同一设备的任何读写操作严格串行，避免请求/响应交叉错乱。
- 不同设备的通道完全独立，可真正并发执行。

---

### 5. 握手流程

协议配置中的 `handshake` 数组定义了连接建立后需执行的一系列初始化步骤。每个步骤包含：

- 请求模板
- 独立的帧解析配置（可覆盖协议全局 `framing`）
- 有效性校验条件

通道建立后顺序执行握手步骤，任意一步失败则标记设备不可用。该机制使 S7 的 COTP 连接等复杂流程完全由配置描述，无需硬编码。

---

### 6. 模板变量与内置函数小结

| 占位符示例 | 说明 |
|-----------|------|
| `{UnitID:X2}` | 取变量 UnitID，格式化为 1 字节大端十六进制 |
| `{StartAddress:X4}` | 取变量 StartAddress，格式化为 2 字节大端十六进制 |
| `{TransactionID:auto:X4}` | 引擎自增计数器，输出 2 字节事务 ID |
| `{Length:calc:X4}` | 自动计算后续字节长度，填入 2 字节 |

---

**核心思想**：网关不包含任何协议细节，只提供通用的模板引擎、帧解析器、并发通道和表达式校验器。通过加载不同的 JSON 配置，即可支持 Modbus、S7 及未来任意二进制协议，真正做到“定义即执行”。
